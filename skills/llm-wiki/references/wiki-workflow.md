# Wiki Workflow

Distilled local-first pattern:

1. Inspect the existing vault or repo layout first.
- Look for raw-note folders, curated history folders, typed entity folders, and index pages.
- Preserve the existing taxonomy before proposing any new structure.
2. Split outputs by job.
- Timeline or chapter pages capture sequence and link to supporting raw notes.
- Canonical entity pages capture stable facts about one person, place, faction, or item.
- Index pages help humans navigate repeated entities.
3. Update canonical pages before creating new ones.
- If an entity already has a page, add the new fact there and link the new note back to it.
- If an entity is missing and clearly recurring, create one canonical page in the existing typed folder.
4. Keep page shapes inspectable.
- Timeline or chapter note: short summary, raw-note links, narrow tables when the repo already uses them.
- Entity page: overview, vital stats, major events, references.
- Index page: one row per canonical entity with summary navigation fields.
5. When proposing or making an update, name the specific supporting source path for each non-obvious fact.
6. Link repeated mentions to the canonical page instead of duplicating the description in every note.
7. When no taxonomy exists, start with the smallest default:
- one timeline or history page
- one index page
- canonical pages only for repeated people, places, or items

Earlier wiki-building work also converged on these habits:
- bin raw notes into the right chapter, arc, or history page before polishing prose
- convert plain mentions into clickable canonical links
- avoid touching editor state or unrelated folders
- keep uncertainty visible instead of inventing lore
